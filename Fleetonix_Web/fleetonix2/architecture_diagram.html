Architecture Documentation

1. System Architecture Overview

The Fleetonix system follows a multi-tier architecture pattern, also known as a layered architecture. This architecture separates concerns into distinct layers, each with specific responsibilities, promoting maintainability, scalability, and security.

The system is deployed on an Apache web server (via XAMPP) and uses PHP as the server-side scripting language, MySQL as the relational database management system, and standard web technologies (HTML, CSS, JavaScript) for the client-side interface.

2. User Interface Layer

The User Interface Layer consists of three distinct interfaces serving different user roles:

Client Web Interface

  • Dashboard for booking overview
  • Booking creation and management
  • Schedule viewing and tracking
  • Real-time driver location tracking
  • Notification center
  • Profile management

Admin Web Interface

  • System-wide dashboard
  • User management (CRUD operations)
  • Booking approval and management
  • Schedule creation and editing
  • Driver performance monitoring
  • Accident report management
  • System notifications

Driver Mobile App

  • Mobile-optimized interface
  • Login and authentication
  • Trip assignment viewing
  • Real-time GPS location updates
  • Trip status updates (start, pickup, dropoff)
  • Accident reporting with photos
  • Vehicle issue reporting

All interfaces communicate with the server using HTTP/HTTPS protocols, with the mobile app using RESTful API endpoints for data exchange.

3. Presentation Layer

The Presentation Layer handles the initial request processing and user interaction:

  • Login Pages: Handle user authentication requests, registration, password recovery, and OTP verification. These pages serve as entry points for all user types.
  • Client Pages: Serve dynamic content for client users, including dashboards, booking interfaces, and schedule views. These pages require client-level authentication.
  • Admin Pages: Provide administrative interfaces for system management. These pages enforce admin-level access control and display system-wide data and statistics.

The presentation layer acts as a controller that routes requests to appropriate application layer modules based on user actions and permissions.

4. Application Layer

The Application Layer contains the core business logic modules that process user requests:

Authentication Module

  • User credential validation
  • Session management
  • Role-based access control (RBAC)
  • Password hashing and verification

OTP Handler

  • OTP code generation
  • OTP storage and expiration
  • OTP verification logic
  • Integration with email service

Booking Management

  • Booking creation and validation
  • Driver assignment logic
  • Booking status updates
  • Booking cancellation

Schedule Management

  • Recurring schedule creation
  • One-time schedule handling
  • Schedule editing and deletion
  • Schedule-to-booking conversion

User Management

  • User CRUD operations
  • Client profile management
  • Driver profile management
  • Account activation/deactivation

Notification System

  • Notification generation
  • Notification delivery
  • Read status tracking
  • Notification filtering

Location Service

  • GPS coordinate reception
  • Location data storage
  • Real-time location retrieval
  • Location history tracking

Accident Reporting

  • Accident report creation
  • Image upload handling
  • Location data association
  • Administrator notification

5. API Layer (RESTful Services)

The API Layer provides standardized RESTful endpoints for mobile application integration:

  • Driver Login API: Authenticates driver credentials, generates OTP, and returns session tokens. Endpoint: POST /api/driver_login.php
  • Location API: Accepts GPS coordinates from mobile devices and stores location data. Provides real-time location retrieval. Endpoints: POST /api/driver_location.php, GET /api/driver_locations.php
  • Trip Management API: Handles trip lifecycle events including trip start, pickup confirmation, dropoff confirmation, return to pickup, and trip completion. Updates booking status in real-time.
  • Accident API: Processes accident reports with details, images, and location data. Creates notifications for administrators. Endpoint: POST /api/driver_accident_report.php
  • Vehicle Issue API: Allows drivers to report vehicle problems that may affect trip execution. Endpoint: POST /api/driver_vehicle_issue.php
  • Address Search API: Provides address autocomplete functionality using external geocoding services. Endpoint: GET /api/address_search.php

All API endpoints implement CORS (Cross-Origin Resource Sharing) policies, input validation, error handling, and return responses in JSON format for consistent client-side processing.

6. Business Logic Layer

The Business Logic Layer contains reusable helper functions and business rules:

  • Client Functions: Contains business logic specific to client operations, including booking statistics, schedule retrieval, and client-specific data processing.
  • Admin Functions: Provides administrative business logic including system statistics, user management operations, and system-wide data aggregation.
  • API Helper: Contains utility functions for API endpoints including request parsing, response formatting, CORS handling, and input validation.
  • Email Helper: Manages email operations using PHPMailer library, including OTP email sending, notification emails, and email template management.

This layer promotes code reusability and separation of concerns, ensuring that business logic is centralized and maintainable.

7. Data Access Layer

The Data Access Layer abstracts database operations from the business logic:

  • Database Connection: Manages MySQL database connections using connection pooling. Handles connection errors gracefully and sets appropriate timezone (Asia/Manila) and character encoding (UTF-8) for internationalization support.
  • Query Handlers: Executes prepared statements to prevent SQL injection attacks. Handles transactions for data integrity and implements proper error handling and logging. All database queries use parameterized statements.
  • Data Validation: Validates input data before database operations, sanitizes user inputs to prevent XSS attacks, and ensures data consistency and referential integrity through validation rules.

This layer uses prepared statements and parameterized queries exclusively to prevent SQL injection vulnerabilities and ensure secure database interactions.

8. Database Layer

The MySQL database stores all system data in normalized relational tables:

  • Users Table: Central user account information including email, hashed passwords (bcrypt), user types (admin, client, driver), account status, and timestamps.
  • Clients Table: Client-specific information linked to user accounts via foreign key, including company details, contact information, and preferences.
  • Drivers Table: Driver profiles linked to user accounts, including vehicle assignments, license information, performance metrics, and availability status.
  • Bookings Table: Booking records including pickup/dropoff locations, dates, times, status (pending, approved, in-progress, completed, cancelled), assigned drivers, client information, and pricing details.
  • Schedules Table: Stores recurring and one-time schedules, frequency settings (daily, weekly, monthly), associated bookings, and schedule status.
  • Notifications Table: Tracks all system notifications, their recipients (linked to users), read status, delivery timestamps, and notification types.
  • Locations Table: Stores historical GPS coordinates from drivers with timestamps, enabling location tracking, route analysis, and real-time position display.
  • Accidents Table: Records accident reports with details, image file paths, location data, driver information, resolution status, and administrator notes.
  • OTP Table: Stores one-time password codes with user associations, expiration timestamps, and usage status for two-factor authentication.

The database implements foreign key constraints to maintain referential integrity, indexes on frequently queried columns (user_id, booking_id, driver_id, etc.) for performance optimization, and a normalized structure (3NF) to minimize data redundancy.

9. External Services Integration

The system integrates with external services to extend functionality:

  • SMTP Server (Gmail): Uses PHPMailer library to send emails through Gmail's SMTP service. Handles OTP emails, password reset emails, and system notifications. Configured with app-specific passwords for secure authentication.
  • Geocoding Service: Integrates with external address search APIs to provide address autocomplete functionality. Converts addresses to coordinates and vice versa for location-based features.
  • GPS Service: Receives GPS coordinates from mobile devices (Android/iOS) through the mobile application. The mobile app uses device GPS hardware to provide real-time location updates to the system.

10. Data Flow Example: Complete Booking Process

To illustrate the architecture in action, here's the complete data flow for a booking process:

  1. Client Request: Client accesses the web interface and navigates to create a new booking.
  2. Address Search: Client enters pickup/dropoff addresses. System calls Address Search API which queries external geocoding service to validate and complete addresses.
  3. Form Submission: Client submits booking form. Data is sent via HTTP POST to client/booking_new.php.
  4. Authentication Check: Presentation layer verifies client session and permissions through Authentication Module.
  5. Business Logic: Booking Management module validates booking data, checks driver availability, and applies business rules.
  6. Database Operation: Data Access Layer executes prepared INSERT statement to create booking record in Bookings Table.
  7. Driver Assignment: System queries Drivers Table for available drivers and assigns appropriate driver based on business rules.
  8. Notification Generation: Notification System creates notifications for assigned driver and administrator, storing them in Notifications Table.
  9. Email Notification: Email Helper sends email notification to driver using SMTP service (Gmail).
  10. Response: Success response is sent back to client interface, which updates to show the new booking.
  11. Driver Action: Driver receives notification via mobile app, views booking details through API, and starts trip.
  12. Location Updates: Mobile app periodically sends GPS coordinates to Location API, which stores them in Locations Table.
  13. Real-time Tracking: Client can view real-time driver location by querying Locations Table through Location Service.

11. Security Architecture

Security is implemented at multiple layers:

  • Authentication Security: Passwords are hashed using bcrypt algorithm before storage. Two-factor authentication (2FA) via OTP adds an extra security layer.
  • Session Security: PHP sessions are used with secure cookie settings. Sessions are regenerated on login to prevent session fixation attacks.
  • Authorization: Role-based access control (RBAC) ensures users can only access resources appropriate to their role (admin, client, driver).
  • Input Validation: All user inputs are validated and sanitized at multiple layers (client-side JavaScript, server-side PHP) to prevent XSS and injection attacks.
  • SQL Injection Prevention: All database queries use prepared statements with parameterized queries, completely preventing SQL injection vulnerabilities.
  • CORS Policy: API endpoints implement proper CORS policies to control cross-origin access and prevent unauthorized API usage.
  • HTTPS Support: System is designed to work with HTTPS for encrypted data transmission (when SSL certificates are configured).

12. Deployment Architecture

The system is deployed using the following infrastructure:

  • Web Server: Apache HTTP Server (via XAMPP) handles HTTP requests, serves static files, and executes PHP scripts.
  • Application Server: PHP runtime environment processes server-side logic, manages sessions, and handles database connections.
  • Database Server: MySQL database server (via XAMPP) stores and manages all persistent data.
  • File System: Server file system stores uploaded files (accident images, documents) and application code.
  • Network: System communicates over HTTP/HTTPS protocols. Mobile apps communicate via RESTful API over HTTP/HTTPS.

The architecture supports horizontal scaling potential - the stateless API design allows API endpoints to be deployed across multiple servers behind a load balancer if needed for high-traffic scenarios.
